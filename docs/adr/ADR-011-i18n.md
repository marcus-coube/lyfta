# ADR-011 — Idiomas: pt-BR, pt-PT e en desde o lançamento

**Status:** Aceito
**Data:** 2026-07-02

## Contexto

Decisão de produto: lançar com português do Brasil, português de Portugal e inglês.

## Decisão

1. **Strings da UI:** ARB + `flutter_localizations`/`intl` com locales `pt_BR`, `pt_PT`
   e `en`. `pt_BR` é a língua-fonte; `pt_PT` herda de `pt` com overrides pontuais
   (ex.: "academia" vs "ginásio", "aluno" vs "aluno/atleta") para não duplicar 100% das chaves.
2. **Conteúdo dinâmico (biblioteca de exercícios):** tabela de traduções
   `exercise_translations (exercise_id, locale, name, instructions)` com fallback
   `pt_PT → pt_BR → en`. A curadoria do ADR-010 já produz os três locales.
3. **Backend:** mensagens de erro da API retornam `code` estável + params; a tradução
   é responsabilidade do cliente. Notificações push são renderizadas pelo servidor no
   locale do destinatário (`users.locale`).
4. **Formatação:** datas, números e moeda sempre via `intl` com o locale do usuário;
   moeda segue a currency do tenant (ADR-007), não o locale.
5. Regra de disciplina: **nenhuma string literal de UI em código** — lint bloqueia.

## Consequências

- Custo contínuo real: toda feature nasce com 3 locales (UI + push + conteúdo).
- O fallback em cascata evita que pt-PT trave lançamentos.
- Novos idiomas (es) ficam baratos: é adicionar locale, não refatorar.
